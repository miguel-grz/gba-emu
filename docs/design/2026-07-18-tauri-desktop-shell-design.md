# Tauri desktop shell — design spec

**Date:** 2026-07-18
**Status:** Approved, ready for implementation plan

## Goal

Package the existing Pocket web frontend as a native desktop application using
Tauri v2, with a native window and a native menu bar whose **File → Open ROM…**
feeds the existing import flow through a native file dialog. The emulator keeps
running as WebAssembly inside the webview — Tauri only provides what a browser
cannot (native window, menu, file dialog).

Scope for this iteration: native shell + native "Open ROM" menu, built and
verified on macOS (the developer's platform). Configuration is kept
cross-platform-friendly, but Windows/Linux builds are out of scope for this pass
(they require their own build host).

## Architecture (Approach A — Tauri as a pure shell)

The WASM core runs in the webview exactly as on the web. IndexedDB, localStorage,
and Web Audio all work in Tauri's webview, so the library, saves, save states,
and controls config carry over with no changes. Tauri adds the native window,
the native menu, and the native file dialog; the "Open ROM" menu item only feeds
ROM bytes into the import flow that already exists.

Rejected alternatives: porting the emulator to native Rust via IPC (massive
rearchitecture, worse per-frame canvas latency, discards the working WASM path);
a hybrid moving ROM/save I/O to the native filesystem (added complexity for no
gain, since IndexedDB already persists).

## Project structure

```
src-tauri/
├── Cargo.toml            # crate "pocket" (bin) — tauri v2, tauri-plugin-dialog, tauri-plugin-fs
├── build.rs              # tauri_build::build()
├── tauri.conf.json       # app / window / bundle / build hooks
├── capabilities/
│   └── default.json      # v2 permissions: core:default + dialog:default + fs read
├── icons/                # app icons (generated by `tauri icon`)
└── src/
    ├── main.rs           # entrypoint → pocket_lib::run()
    └── lib.rs            # setup: native menu + on_menu_event emitting "menu://open-rom"
```

- The root `Cargo.toml` workspace `members` gains `"src-tauri"` (single lockfile).
  The existing `[profile.release] lto = true` applies.

## 1. Tauri configuration (`src-tauri/tauri.conf.json`)

- `productName: "Pocket"`, `identifier: "dev.pocket.gba"`, `version: "0.1.0"`.
- `build.frontendDist: "../dist"`, `build.devUrl: "http://localhost:5180"`.
- `build.beforeDevCommand: "npm run dev"`.
- `build.beforeBuildCommand: "npm run wasm && npm run build"` — builds the WASM
  package before `vite build`, resolving the `web/pkg` dependency.
- `app.windows[0]`: `title: "Pocket"`, `width: 1200`, `height: 820`,
  `minWidth: 940`, `minHeight: 640`, `resizable: true`.
- `bundle.active: true`, `bundle.targets: "dmg"` (macOS). Config stays
  cross-platform; other targets are not built this pass.
- `app.withGlobalTauri: false` — the frontend uses the `@tauri-apps/api` npm
  package, not the injected global.

The Vite dev server currently runs on port **5180** (observed). `devUrl` must
match. If Vite is configured to a fixed port, keep them in sync; otherwise pin
Vite's port to 5180 in `vite.config.ts` so `devUrl` is stable.

## 2. Native menu + "Open ROM" bridge

Reading the ROM lives in JS (next to the existing import code); Rust only builds
the menu and forwards the click, keeping the IPC surface minimal.

**Rust (`src-tauri/src/lib.rs`):**
- In Tauri's `setup`, build a native menu with the v2 menu API:
  - **File** submenu: a `MenuItem` with id `open-rom`, text "Open ROM…",
    accelerator `CmdOrCtrl+O`; a separator; the predefined Quit item.
  - On macOS, include the predefined app submenu (About/Quit) and an Edit
    submenu so standard shortcuts feel native.
- `on_menu_event`: when the menu id equals `open-rom`, call
  `window.emit("menu://open-rom", ())`. It touches no files.
- Register `tauri_plugin_dialog::init()` and `tauri_plugin_fs::init()`.
- `main.rs` calls `pocket_lib::run()`.

**Frontend (`src/lib/desktop.ts`, new):**
- `isDesktop(): boolean` — detects Tauri via `"__TAURI_INTERNALS__" in window`.
- `initDesktop(onOpenRom: (name: string, bytes: Uint8Array) => void): Promise<() => void>`
  — if not desktop, returns a no-op cleanup. If desktop, subscribes to the
  `menu://open-rom` event (via `@tauri-apps/api/event`). On the event:
  1. Open the native dialog with `open()` from `@tauri-apps/plugin-dialog`,
     filter `{ name: "GBA ROM", extensions: ["gba", "bin"] }`, `multiple: true`.
  2. For each chosen path, read bytes with `readFile(path)` from
     `@tauri-apps/plugin-fs` → `Uint8Array`.
  3. Derive the base name from the path and call `onOpenRom(name, bytes)`.
  Returns the `unlisten` function for effect cleanup.
- All `@tauri-apps/*` imports are dynamic (`await import(...)`) inside the
  `isDesktop()` guard, so a pure web build never loads the plugins or breaks.

**Permissions (`src-tauri/capabilities/default.json`):** `core:default`,
`dialog:default`, and the `fs` read permission. Tauri v2 grants read access to
paths returned by the dialog, so no broad filesystem scope is opened.

## 3. Frontend changes (`src/App.tsx`)

Extract the import core from `addRom` into a reusable function:

```ts
const importRom = useCallback(async (name: string, bytes: Uint8Array) => {
  setError(null);
  setBusy(name);
  try {
    await addGame(name, bytes, { title: resolveTitle(bytes, name), cover: coverUrl(bytes) });
    await refreshGames();
    const thumb = await generateThumbnail(bytes);
    await setThumbnail(name, thumb);
    await refreshGames();
  } catch (e) { setError(String(e)); }
  setBusy(null);
}, [refreshGames]);

const addRom = useCallback(async (file: File) => {
  importRom(file.name, new Uint8Array(await file.arrayBuffer()));
}, [importRom]);
```

Wire the native menu with an effect that runs once on mount:

```ts
useEffect(() => {
  let cleanup = () => {};
  initDesktop(importRom).then((fn) => { cleanup = fn; });
  return () => cleanup();
}, [importRom]);
```

On the web, `isDesktop()` is false, so this is a no-op. The UI is otherwise
unchanged; drag-drop, the file input, and the native menu all funnel through
`importRom`, reusing the same library/thumbnail/title/cover pipeline.

## 4. Dependencies

- JS devDependency: `@tauri-apps/cli`. JS runtime: `@tauri-apps/api`,
  `@tauri-apps/plugin-dialog`, `@tauri-apps/plugin-fs`.
- `package.json` script: `"tauri": "tauri"`.
- Rust (`src-tauri/Cargo.toml`): `tauri` v2 (features as scaffolded),
  `tauri-plugin-dialog` v2, `tauri-plugin-fs` v2, `serde`/`serde_json` as needed;
  build-dependency `tauri-build` v2.

## Build & dev

- **Dev:** `npm run tauri dev` — Tauri runs `npm run dev` (Vite on :5180), opens
  the native window at `devUrl` with HMR.
- **Build:** `npm run tauri build` — runs `npm run wasm && npm run build`, then
  bundles the `.app`/`.dmg` from `dist`.

## Error handling

- `initDesktop`: guarded by `isDesktop()`; dynamic imports mean a web build never
  touches Tauri APIs. Dialog cancel (returns `null`) is a no-op. `readFile`
  errors are surfaced through the existing `setError` path in `importRom`
  (wrap the desktop read in try/catch that calls the same error channel).
- `importRom` keeps the existing try/catch, so a bad ROM shows the same error UI
  regardless of source.

## Testing / verification

The Rust core (`gba-core`) does not change — no new `cargo test` there. The new
Rust is `src-tauri` glue (menu + event emit), which has no unit-test seam worth
adding. Verification is build + runtime:

- **Frontend still builds:** `npx tsc --noEmit` clean after the `importRom`
  refactor and the new `desktop.ts`.
- **Web path unregressed:** in the browser preview, drag-drop / file-input import
  still works (the refactor must not break the existing flow).
- **Tauri compiles:** `cargo build` in `src-tauri` succeeds (first build is slow —
  it compiles many WebKit-binding crates).
- **Tauri dev starts:** launch `npm run tauri dev` in the background and confirm
  from logs that Vite starts and Tauri creates the window with no errors.
- **Explicit environment limit:** this environment can drive the localhost
  webview preview but cannot screenshot or click a native macOS window. The final
  visual check — open the native window, use File → Open ROM…, see a game run —
  is confirmed by the developer running `npm run tauri dev`, or inferred from
  logs. This limit is stated in the plan rather than claimed as passed.

## Risks

- System dependencies for Tauri may be missing; on macOS, Xcode Command Line
  Tools + Rust usually suffice (both present).
- The WASM `?url` import must resolve correctly in the packaged bundle — verified
  during `tauri build`.
- First `tauri build`/`dev` compile is slow (minutes); accounted for in the plan.

## Out of scope (this iteration)

- Windows/Linux packaging (config is left ready; builds need those hosts).
- Native filesystem storage for ROMs/saves (IndexedDB is retained).
- Auto-update, code signing/notarization, deep links, tray icon.
- Native drag-drop onto the window (the in-webview drag-drop already works).
